# 성능 및 유지보수 전수 점검 보고서

- 점검일: 2026-07-24
- 기준 커밋: `dd49e6678818ad008539e2ad5e8128ffcae5258d`
- 대상: 단일 정적 애플리케이션 `index.html`
- 측정 환경: 로컬 HTTP 서버, Chromium 141, 데스크톱/모바일 뷰포트
- 원칙: 기능과 UI를 바꾸지 않고, 측정으로 확인된 안전한 변경만 적용

## 요약

현재 구조에서 실제로 반복 비용이 확인된 부분은 태그 UI를 다시 만드는 `initApp()` 경로였다. 태그마다 같은 입력 장치 판정을 세 번 반복하던 코드를 한 번만 계산하도록 바꿨다. 4배 CPU 제한 조건의 독립 측정 두 세트에서 `initApp()` 중앙값은 평균 87.75ms에서 82.30ms로 약 6.2% 줄었고, 레이아웃을 강제로 확정한 측정은 평균 89.05ms에서 78.45ms로 약 11.9% 줄었다.

반면 22,000줄이 넘는 단일 파일 자체는 유지보수 위험이 크지만, 현재 로컬 실행에서 파싱과 초기 스크립트 실행이 주 병목이라는 근거는 나오지 않았다. 파일 분리는 로딩 순서, 전역 함수, 인라인 이벤트 핸들러의 결합으로 회귀 위험이 큰 반면 즉시 얻을 수 있는 성능 효과는 불확실하므로 이번 변경에서는 보류했다.

## 변경 전후 측정

4배 CPU 제한을 적용하고 기준/변경 버전을 각각 두 번 독립 실행했다. 아래 값은 각 실행의 중앙값 두 개와 그 평균이다.

| 시나리오 | 기준 측정 | 변경 측정 | 평균 변화 |
| --- | ---: | ---: | ---: |
| `initApp()` | 84.8 / 90.7ms | 83.5 / 81.1ms | 87.75 → 82.30ms, 약 6.2% 감소 |
| `initApp()` + 강제 레이아웃 | 89.1 / 89.0ms | 72.4 / 84.5ms | 89.05 → 78.45ms, 약 11.9% 감소 |
| Inpaint 2-frame 화면 전환 | 247.8 / 288.6ms | 225.1 / 225.7ms | 268.20 → 225.40ms, 약 16.0% 감소 |

Classic, Studio, Img2Img, Anima 화면은 작은 개선 또는 측정 잡음 범위였다. 따라서 전체 앱이 일괄적으로 빨라졌다고 확대 해석하지 않고, 반복 태그 생성 경로에서 확인된 개선으로 범위를 제한한다.

최종 변경 버전의 콜드 부트 중앙값은 다음과 같다.

| 항목 | 값 |
| --- | ---: |
| `domInteractive` | 202.2ms |
| Script duration | 39.004ms |
| DOM 노드 | 5,125 |
| 이벤트 리스너 | 2,177 |

초기 노드와 리스너 수는 변경 전과 동일해 이번 최적화가 기능 구조를 바꾸지 않았음을 확인했다.

## 22,000줄 단일 파일 분리 판단

파일 분리는 장기적으로 코드 탐색, 소유권 분리, 캐시 효율과 테스트 격리에 도움이 될 수 있다. 그러나 현재 코드는 전역 상태, 인라인 이벤트 핸들러, 선언 순서와 즉시 실행 코드가 서로 결합되어 있다. 단순히 CSS와 JavaScript를 외부 파일로 옮기면 다음 위험이 생긴다.

- 스크립트 로딩·실행 순서 변경
- 전역 함수 노출 시점 변경
- 인라인 핸들러가 참조하는 이름 누락
- 초기화 타이밍과 DOM 준비 시점 차이
- 배포 캐시가 HTML과 자산의 서로 다른 버전을 섞는 문제

이번 측정에서는 단일 파일 파싱이 주 병목이라는 근거가 없었으므로, 성능 최적화를 이유로 한 대규모 분리는 비용 대비 이득이 부족하다. 향후 분리한다면 먼저 전역 API 목록과 화면별 기능 계약 테스트를 만들고, CSS 또는 독립 유틸리티처럼 결합도가 낮은 영역부터 단계적으로 진행하는 것이 안전하다.

## 보류한 최적화 후보

다음 항목은 잠재 효과가 있으나 현재 근거 또는 안전성이 부족해 적용하지 않았다.

- 태그별 drag 이벤트를 이벤트 위임으로 변경: 초기 리스너 수를 줄일 수 있지만 드래그 경계와 모바일 포인터 동작 회귀 위험이 큼
- 전체 화면 DOM의 영구 유지 또는 부분 업데이트: 재생성 비용을 줄일 수 있지만 화면별 격리 상태와 초기화 계약이 달라질 수 있음
- 모든 `localStorage` 저장에 dirty tracking 또는 debounce 적용: 현재 직렬화 비용은 미미하며 저장 시점 변경이 복구 동작을 바꿀 수 있음
- 전역 상태를 모듈로 일괄 전환: 유지보수성에는 유리하지만 현재 성능 병목 해결과 직접 연결되지 않고 회귀 범위가 큼
- 이름만 보고 추정한 미사용 함수 제거: 외부·인라인 호출 가능성을 배제할 수 없는 공개 전역은 유지
